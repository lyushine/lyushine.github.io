我个人所理解的浮动与清除（闭合）方案

在我们进行页面重构（css编写）的时候是避免不了使用浮动。CSS布局方式主要是浮动、定位和基于正常文档流的布局。而浮动布局是这3种中使用频率最高的一种。个人认为能理解浮动,CSS基本布局完全不是问题，能很好的理解浮动及其清除（闭合）浮动思想，那么CSS的兼容性解决开发成本将大大的减少。总之，浮动是CSS布局的大门，要想掌握CSS布局必须要掌握浮动。

浮动到底是什么东西？

1.1 浮动具有包裹性

我们写一个带颜色的DIV（以块级元素示例）。

HTML:
<div>这里是没有浮动的元素一</div>
CSS:
.noFloat{background:#939}

 

正常文档流的块级元素独占一行。

下面我们给两个DIV都设置左浮动。

HTML:
<div>这里是左浮动的元素一</div>
CSS:
.leftFloat{background:#939; float:left}

 

我们可以看到这个浮动元素的宽高刚好为内部的内容的宽高，也就是说浮动具有包裹性把里面的元素包裹起来了。

我们知道，行内元素是不可以设置宽高的，也就是说设置宽高是无效，上面已经知道浮动可以包裹元素，那么这个包裹可否设置宽高？
下面写一个带宽且浮动的行内元素

HTML:
<span>这里是左浮动的元素一</span>
CSS:
leftFloat{background:#939; float:left; width:200px; height:200px}

 

可以应用该属性为行内元素设置宽高。

看到对行内元素的宽高设置有效，那么得出 **浮动具有包裹性可以为所有浮动的元素设置宽高。**

1.2 浮动具有方向性

float有三个值left、right、none，分别代表左浮动、右浮动、不浮动。
我们写一组左浮动，中间查入文字元素，以此为例。

<div>
<div>这里是左浮动</div>
<div>这里是左浮动</div>
<div>这里是左浮动</div>
这里是文字
<div>这里没有换行在有限的空间里挤</div>
<div>这里是左浮动</div>
<div>这里是左浮动</div>
</div>
CSS:
.wrap{width:400px; margin:0 auto}
.leftFloat{background:#939; float:left; width:100px; height:100px}

我们发现在第三个浮动后面写了文字占据了一定的高度和宽度，而第四个浮动元素竟然在匿名文字（这里是文字）下，而第五六浮动元素却换行了，在第四个浮动元素下继续浮动。

发挥思维想象一下，得出：

浮动元素的左浮动类似一个元素从无限远的最右上角一直在找能够容纳自己展现的最左上角的位置并且是以页面元素的右边界开始尝试换行。
右浮动从最左上角开始尝试，类似。

有了方向性我们才可以任意的对页面元素进行灵活布局。

我们仅对图片设置了左浮动就实现了文字环绕图片。我们知道文字是基于行框的，行框不会让内容成块显示，而是排成一排。假如图片没设置浮动，则如下图：

此时文字和图片都是基于行框的，因为行框的内容是排成一排显示的，所以图片和第一行文字对齐。如果想要一张图片和多行文字对齐，唯一的方式是破坏正常的行框。

我们发现，图片浮动后就实现了文字环绕图片，图片浮动后破坏了图片的行内框从而破坏了行框，一旦图片失去了行框特性就无法和文字排在一排了，跟随自己的浮动方向靠边展示。

那么我们说 浮动破坏了行框。

虽然浮动破坏了浮动自身元素的行框，但是浮动元素依旧存在一个自己的躯体占位置。

我们知道绝对定位是脱离文档流的，而浮动元素是依旧在文档中的。

那么

1、浮动导致破坏行框，破坏了行框并不等于没有宽高（包裹性），从而使图片不和一排文字对齐

2、图片存在于文档流，那么就会占位。

FF下不正常，给外套容器设置的边框紧缩在一起处于顶部位置，外套容器无法知道自己内部的元素有多高从而无法高度自适应。当一个元素是浮动的，如果没有清除（闭合）浮动时，其父元素不会包含这个浮动元素，因为此时浮动元素从文档流中脱离。

注：IE6 下正常是因为父容器设置了width从而触发了IE的haslayout所以正常。

是浮动导致了该BUG，浮动不是完美的能引起很多非正常显现而导致浏览器不兼容，与浮动与之对应的是清除（闭合）浮动。清除（闭合）浮动能解决浮动引起的问题。

为了解决浮动引起的问题，我们要清除（闭合浮动）。

我们可以有一个意识，浮动元素或者其周围出现不兼容错位现象，要思考是不是由浮动元素引起的并且尝试清除（闭合）浮动。

清除（闭合）浮动的方法？

CSS中有一个属性clear其值有left right both，clear:left代表清除左边的浮动,clear:right代表清除右边的浮动，clear:both清除两边的浮动，一下解决方案都是基于该属性而实现的。

 

我把应用于父元素的解决浮动问题方法称为闭合浮动，因为应用于父元素就好像把父元素及其其中的内容与其他元素间隔开来，封闭起来。

而使用单独一个标签来清除浮动的方法称为清除浮动。

 

首先我们介绍一下有些时候clear的清除浮动方式对IE6无效，其原因是由于系统没有触发IE6的haslayout属性而无法正常的绘制图形GG。这个时候我们需要帮IE6一把，手动触发haslayout属性从而达到需求的效果。

触发IE6 haslayout的属性有width height position(relative absolute) zoom。

我个人倾向于使用 zoom:1触发ie6 haslayout ，因为这个属性是缩放属性，值为1就是按原大小缩放，所以对页面没任何可见影响。安全绿色！

一、 使用br标签的clear属性清除浮动

<div class=“father”>
   <div class=“”float_left””></div>
   <div class=“”foat_right””></div>
</div>

 

将
放在浮动元素之后而清除浮动。

该方式对IE无效,如果父容器没设置过width height position(relative absolute) zoom属性。

再触发一下haslayout

.father{zoom:1}

这种方式增加了一个额外的标签而且还要配合zoom:1才能达到效果,在父容器设置过width height position(relative absolute) 属性的时候不需要配合zoom:1。

二、增加一个div使用css的clear属性清除浮动

<p align=“left”>.clear {clear:both}</p>
   <div class=“father”>
   <div class=“”float_left””></div>
   <div class=“”foat_right””></div>
</div>

 

将


放在浮动元素之后而清除浮动。

该方式对IE无效,如果父容器没设置过width height position(relative absolute) zoom属性。

再触发一下haslayout

.father{_zoom:1}

这种方法类似br的方式但优于br的方式，因为在页面重构过程中基本要求XHTML和CSS样式相分离，增加了一个额外的标签。

 

一般情况下，父容器的width值是会设定的，所以一般情况下，使用上述两种方式不需要使用zoom:1从而这用在浮动元素后加入清除浮动的元素就行了，使用也较为方便，但是增加了一个额外的没有用的元素。

三、使用overflow .cd方法闭合浮动

使overflow属性来清除浮动有一点需要注意，overflow属性共有三个属性值：hidden,auto,visible。可以使用hiddent和auto值来清除浮动，但不要使用visible值(visible只对IE浏览器有效)。基于IE 6 的haslayout 问题，我们定义一个单独的class应用于父元素而闭合浮动(使用auto可能会产生滚动条)。

.cd{overflow:hidden; _zoom:1}

 

将该class用于内部有浮动的父元素就行了。

 

优点：简单有效，代码简短，没有增加额外标签，便于公用.

缺点：有截断，如果父元素内部有绝对定位或者其他超出父元素的元素时，并且会截断CSS3的一些效果，有点暴力！

 

四、 使用伪类clearfix方法闭合浮动

首先介绍一下:after和:before伪类。在偏向标准的浏览器中，可以通过:after 和:before在标签元素的后面和前面显示更多的信息。，:after和:before相当于在元素前面和后面插入两个行内元素块。

因为:after和:before是行内的所以使用display:block块化，行内元素是不可以清除浮动的，visibility:hidden将元素隐藏但占据位置，height:0将元素高度设为零使不影响布局,clear:both闭合浮动，content 必须填写不为空的任意值，:after :before所“生成”的元素才存在,zoom:1 触发IE haslayout。

.clearfix:before,
.clearfix:after {
content: “.”;
display: block;
height: 0;
visibility: hidden;
}
.clearfix:after {clear: both;}
.clearfix {zoom: 1;}

其实只使用clearfix:after就可以达到清除浮动的效果，但只使用clearfix:after时在跨浏览器兼容问题会存在一个垂直边距叠加的bug。
优点：没有增加额外的标签，便于公用。
缺点：相比.cd方法代码较长
从最初学习CSS开始我是使用


这样的方法的，后来使用.cd{overflow:hidden; zoom:1}方法，.cd方法也不失为一种好方法，简单有效，就是有出现截断的风险性，以前在使用.cd闭合浮动时基本没出现过错误。.clearfix方法虽然我没广泛使用过，但是经测试觉得这种方法更为绿色可行，便于重用。
清除（闭合）浮动： 在以上多种清除（闭合）浮动的里，基本上都是使用最原始的clear:both方法，clearfix更是


和
的变身，只是利用了偏向浏览器的伪类“隐藏元素”。在IE 6 – 7里，我们还需要触发其 haslayout。最常用的安全绿色无副作用的是zoom:1。

几个CSS陷阱和技巧

几个CSS技巧

一、按钮间隙

在input type=”submit”或者button中出现：
/ 移除额外的边距 IE / （这个我还没测出来）

.button{
overflow: visible;
}
/* 移除额外的内部间距 Firefox */ （经过去除后，和IE表现一样，没去时我测的相差1px高） 
.button::-moz-focus-inner{
border: 0;
padding: 0;
}

二、使用伪元素

在行内替换元素(input img)中是不可以使用 :before 、:after等为元素的,因为行内替换元素
是没有“内容”的，因此伪元素不能使用。

注意:before和::before的区别，::before是新写法IE8不识别,所以为了兼容尽可能的使用旧写法
:before。

三、在整个BODY中使用渐变背景

CSS3 渐变是会repeat的，使用下面的代码修正：

html {
height: 100%;
}
body {
height: 100%;
margin: 0;
background-repeat: no-repeat;
background-attachment: fixed;
}

四：在渐变上使用伪状态转换

1、使用background-position 在：hover上，只是利用了轴的偏移。例子 http://twitter.github.com/bootstrap/

a{
background: linear-gradient(orange, red) repeat-x;
display: inline-block;
padding: 20px;
}
a:hover{
background-color: red;
background-position: 0 -15px;
transition: background-position .1s linear;
}
2、使用一种颜色，而仅仅渐变他的不透明度，:hover状态使用渐变的那一种颜色。 
a{
background-color: orangered;
background-image: linear-gradient(top, rgba(255,255,255,.3), rgba(255,255,255,0));
transition: background-color .1s linear;
display: inline-block;
padding: 20px;
}
a:hover{
background-color: red;
}

五、Inline-block 间隙

当使用inline-block元素并列的时候会产生几像素的间距，并且各浏览器的间距还不同。
需要怎么样呢？
比如：

<ul>

IE6 BUG诡异的行内替换元素设置浮动并设置text-indent：xx px 影响相邻元素布局

E6 BUG诡异的行内替换元素设置浮动并设置text-indent：xx px 影响相邻元素布局

一个奇怪的现象：一个DIV里面有两个各元素，一个元素右浮动，浮动的元素设置了个text-indent,而另外一个元素却仅在IE6下产生了和该元素一样大小的类padding-left间距值。

发现原型：

<input type=“button” value=“搜索” /> <a href=“#”>啊啊啊啊</a></div>
<span style=“color: #ff0000;”>在input标签中设置text-indent:20px后，会发现 a标签 在IE6中会出现一个类似左padding值等于20px。</span>

经过一系列测试发现问题产生的条件：

1、 一个内联替代元素【input,img等】和一个内联元素【含内联可替换和内联不可替换元素】相邻

2、 内联可替代元素浮动，内联元素不浮动

3、 内联可替代元素设置 text-align: xx px;

导致的结果：内联元素出现类似padding-left:xx px 的左距离 其中xx 等于 内联可替代元素的text-indent数值。

已知解决方式：

1、 将内联元素也设置浮动

2、 后一元素使用绝对定位

3、 用padding-left替代text-align

未知求解：未知其产生BUG的原理从而找出根本的解决方式，上述解决方式个人觉得较为牵强。

以上探究纯属个人测试个人观点，如有不对欢迎指正。

巧妙解决使用DD_belatedPNG解决IE6PNG透明问题引发的relative定位冲突

在帮助同学完成湖南科技大学ACM判题系统的时候，由于需要解决PNG在IE6下的透明引入并使用了DD_belatedPNG.js(其他方式有弊端无法满足需求)。而在这过程中发现在 使用 DD_belatedPNG. 的元素上进行 position:relative; 会发生错位，而我又必须使用position:relative,这里是为了使用Z-INDEX实现满足需求的罩，z-index属性必须和position:relative或者position:absolute同时使用才奏效实现对应的层级遮罩关系。

要求效果：

下面的主体内容要遮罩中间奖牌的一部分，那么奖牌和下面的主体部分都需要设置z-index和position:relative,那么必然导航即使用DD_belatedPNG又使用position:relative就会引起错位，这不在IE6下飘顶了。

情形：使用DD_belatedPNG和position:relative会引起错位。

那么如何解决这个错位了？在本需求中DD_belatedPNG(处理PNG透明并允许图片定位)和position:relative(触发z-index有效)都是必须的。

在一番纠结之后，我突然发现IE6下我没设置position:relative的话z-index也是有效的。我表怀疑态度Z-INDEX在IE6下不需要position:relative等触发，然后再自己写DEMO考证，IE6下是需要触发的。那么继续找线索，最后找到导航元素（含奖牌那）是使用了DD_belatedPNG.js才导致不需要position:relative触发而使z-index有效的。

那么既然IE6下直接使用 position:static 结合 DD_belatedPNG就可以使z-index有效,而ie6下使用position:relative反而错位，那么写一个HACK，IE6下导航元素position:static，其他浏览器position:relative(触发z-index有效)。

分析问题，利用DD_belatedPNG对不同浏览器不同position的不同效果和Z-INDEX的影响结合，解决了这个问题，个人认为这是一个较为好的解决方案。

虽然用了一个HACK,关于HACK的使用。最开始学习CSS的时候利用过点，后来学习过程中知道提倡NO HACK，WEB标准什么的就走上了NO HACK 鄙视使用HACK之路，再后来思维又改变觉得在代码精简的基础上可以使用CSS HACK或者自己权衡各种需求可维护性等的情况下适当使用CSS HACK。

在这里我个人认为使用一个CSS HACK是较为好的，为什么呢？本来 DD_belatedPNG 也是写了一个HACK判断，只在IE6下使用，而我也只在IE6下_position:static 这样无可厚非。 现在的历史环境下 ，HTML5 CSS3的部分支持下 不是写各种CSS HACK，让优秀的浏览器获的更好的用户体验么？而低版本的浏览器在效果体验率差依旧满足人们日常的使用。

记得在边锋实习期间，也碰到了同样这个问题，那个时候没有好好分析研究。采用的是改变DIV的顺序，而使用margin-top实现遮罩，不得已绕道避开了DD_belatedPNG引起的定位错误，没有使用z-index，这次遇到了同一个问题终于没有放过它在研究实践中发现特征用一个个人认为更为普遍性的方法解决。

而在这个帮同学ACM系统（设计+前端）中的需求而言，学校的内部使用项目，没有多大得用户量，不需要别人进行程序维护，那么我在这里没有对其进行优化兼容即可，满足开发使用需求即可。

哎，最近老想出个什么作品，具有代表性的作品，可惜既想不出也还做不出，还是先好好打好基础。

稍微发下代码：
