CSS七种方法实现分割中线

在项目开发过程中的会员中心页碰到了一个中线，然后发散思维，要求中间的线自适应高度，想了七种方案。

如图效果：

 

方案一：左边标签设置左边框，右边标签设置左边框，利用负margin边框重叠

.middle_line_wrap1 .row {
    padding-left:100px;    /* 保证dd的位置，不管有没有dt，dd的位置都距离左边这么远 */
    zoom:1         /*闭合浮动，针对IE*/
}
.middle_line_wrap1 .row:after {    /*闭合浮动针对支持该伪类的现代浏览器*/
    content:"\20";
    clear:both;
    visibility:hidden;
    display:block;
    height:0
}
.middle_line_wrap1 dt {
    width:99px;
    margin-left:-100px; /* 保证dd的位置，不管有没有dt，dd的位置都距离左边这么远 */
    float:left;        /*让元素可以定宽，而又保持布局，不可以用inline-block,inline-block会引起文字元素间的莫名间隙*/
    text-align:right;
    position:relative;
    z-index:1;        /*兼容IE6，为什么需要使用，原因不详*/
    border-right:1px solid #CECECE
}
.middle_line_wrap1 dd {
    border-left:1px solid #CECECE;
    float:left;       /*解决dd元素边框不识别多行的总高，若没有，则本元素边框为一行元素的高而不是多行*/
    padding-left:10px;
    margin-left:-1px;
    position:relative;
    z-index:1px;    /*兼容IE6，为什么需要使用，原因不详*/
}
<div class="middle_line_wrap1"><dl class="row"><dt>姓名：</dt><dd>LYuShine</dd></dl></div>

很“保险”（使程序输出的时候左边和右边任意一边内容为空甚至标签不输出都正常显示），多了一个多余标签。

 

方案二：设置父容器左边框，父容器margin-left,子元素父margin拉回。

}

    padding-left:100px;    /* 保证dd的位置，不管有没有dt，dd的位置都距离左边这么远 */

    content:"\20";
    clear:both;
    visibility:hidden;
    display:block;
    height:0
}
.middle_line_wrap2 dt {
    width:100px;
    float:left;
    text-align:right;
    margin-left:-200px; /* 保证dd的位置，不管有没有dt，dd的位置都距离左边这么远 */

    float:left;
    margin-left:-100px;

最精简全兼容CSS代码实现1像素圆角

看到了莫有，1像素的圆角，代码精简哦，兼容主流浏览器，当然比不上CSS3的border-radius精简哦。



今天在群里看到一个分享代码写的1px圆角，然后用那代码写了下，精简是精简，但是不兼容。于是稍微修改完善了一下改兼容了。 以前还从没有用过 margin: -1px 0; 因为写过这样的不兼容，解决方法还是有的，下面就是代码了。

.radius,.radius_inner{border-width:1px; border-color:red;}
.radius {display: inline-block; border-style:none solid;}
.radius_inner {border-style:solid none; float:left; margin: -1px 0; position: relative; *width:100%;}

<div class=“radius”>
<div class=“radius_inner”>看到了莫有，1像素的圆角，代码精简哦，兼容主流浏览器，当然比不上CSS3的border-radius精简哦。</div>
</div>
以前一般写兼容的圆角用图片滑动门使用较多，有个时候用下N个标签嵌套，1像素的圆角也是最近工作的时候才碰到，刚开始看着好纠结的，这么一点点设计师为啥么不直接用直角啊，搞的我刚开始觉得是出图不严谨导致的误差效果，因为以前确实发过来的设计图有个时候有的地方不是很严谨。好吧，下次有1px的圆角又要求兼容IE系列的话就直接用这个代码丢进去了。

css分割线的几种写法--源自新浪微博的代码的思考

方法一** 单标签奇葩实现：**

.demo1{padding:0 30px 0;
       margin:7px 0;
       line-height:1px;
       border-left:100px solid #ccc; /*长度的刚性局限性*/
       border-right:100px solid #ccc; /*长度的刚性局限性*/
}

<div class="demo1">5分钟前，你看到这分钟</div>

.demo2{height:1px; border-top:1px solid #ccc; text-align:center;}

        top:-7px;
        margin-top:7px;

       }

<div class="demo2"><span>5分钟前，你看到这分钟</span></div>

四、 浮动与相对定位孕育的神奇的未知宽度水平居中


大多数时候我们用惯了 width:xxx px + margin:0 auto,而如何实现未知宽度水平居中了。这里有个神奇的方案。

结构如下：

<div class=“wrap”>
<div class=“member_paging_wrap”>
<div class=“member_paging”>

<a href=“#”>1</a><a href=“#”>2</a><a href=“#”>3</a><span>…</span><a href=“#”>4</a><a href=“#”>5</a><a href=“#”>6</a>

</div>
</div>
<div class=“member_paging_wrap”></div>
</div>
实现本方案的核心代码：

.member_paging_wrap{float:left; position:relative; left:50%; }

.member_paging{ float:left; position:relative; right:50%; }

分页的外套父容器.member_paging_wrap 通过float:left获得的其子容器 member_paging加上的member_paging作为宽度为186px。子容器.member_paging 是靠.member_paging{ float:left; }获得宽度的。(这里利用的是浮动的包裹性，浮动元素里有内容未本身没定宽度将识别子容器宽度并获得其宽度)。

然后通过相对定位 position:relative; left:50%，则.member_paging_wrap距离左边的50%是其外容器.wrap/2的值为480px。(position:relative **的百分比是按其父容器的宽度来确定的）**。

这样一来member_paging_wrap的左边缘在整个容器.wrap的中线上来。当然这还不是我们想要的。

我们要将里面的分页.member_paging 往左拉其分页包裹外容器的一半距离才能实现居中。

那么继续

.member_paging{ float:left; position:relative; right:50%; }

同样的原理，居中了。

我个人所理解的浮动与清除（闭合）方案

在我们进行页面重构（css编写）的时候是避免不了使用浮动。CSS布局方式主要是浮动、定位和基于正常文档流的布局。而浮动布局是这3种中使用频率最高的一种。个人认为能理解浮动,CSS基本布局完全不是问题，能很好的理解浮动及其清除（闭合）浮动思想，那么CSS的兼容性解决开发成本将大大的减少。总之，浮动是CSS布局的大门，要想掌握CSS布局必须要掌握浮动。

浮动到底是什么东西？

1.1 浮动具有包裹性

我们写一个带颜色的DIV（以块级元素示例）。

HTML:
<div>这里是没有浮动的元素一</div>
CSS:
.noFloat{background:#939}

 

正常文档流的块级元素独占一行。

下面我们给两个DIV都设置左浮动。

HTML:
<div>这里是左浮动的元素一</div>
CSS:
.leftFloat{background:#939; float:left}

 

我们可以看到这个浮动元素的宽高刚好为内部的内容的宽高，也就是说浮动具有包裹性把里面的元素包裹起来了。

我们知道，行内元素是不可以设置宽高的，也就是说设置宽高是无效，上面已经知道浮动可以包裹元素，那么这个包裹可否设置宽高？
下面写一个带宽且浮动的行内元素

HTML:
<span>这里是左浮动的元素一</span>
CSS:
leftFloat{background:#939; float:left; width:200px; height:200px}

 

可以应用该属性为行内元素设置宽高。

看到对行内元素的宽高设置有效，那么得出 **浮动具有包裹性可以为所有浮动的元素设置宽高。**

1.2 浮动具有方向性

float有三个值left、right、none，分别代表左浮动、右浮动、不浮动。
我们写一组左浮动，中间查入文字元素，以此为例。

<div>
<div>这里是左浮动</div>
<div>这里是左浮动</div>
<div>这里是左浮动</div>
这里是文字
<div>这里没有换行在有限的空间里挤</div>
<div>这里是左浮动</div>
<div>这里是左浮动</div>
</div>
CSS:
.wrap{width:400px; margin:0 auto}
.leftFloat{background:#939; float:left; width:100px; height:100px}

我们可以有一个意识，浮动元素或者其周围出现不兼容错位现象，要思考是不是由浮动元素引起的并且尝试清除（闭合）浮动。

清除（闭合）浮动的方法？

CSS中有一个属性clear其值有left right both，clear:left代表清除左边的浮动,clear:right代表清除右边的浮动，clear:both清除两边的浮动，一下解决方案都是基于该属性而实现的。

 

我把应用于父元素的解决浮动问题方法称为闭合浮动，因为应用于父元素就好像把父元素及其其中的内容与其他元素间隔开来，封闭起来。

而使用单独一个标签来清除浮动的方法称为清除浮动。

 

首先我们介绍一下有些时候clear的清除浮动方式对IE6无效，其原因是由于系统没有触发IE6的haslayout属性而无法正常的绘制图形GG。这个时候我们需要帮IE6一把，手动触发haslayout属性从而达到需求的效果。

触发IE6 haslayout的属性有width height position(relative absolute) zoom。

我个人倾向于使用 zoom:1触发ie6 haslayout ，因为这个属性是缩放属性，值为1就是按原大小缩放，所以对页面没任何可见影响。安全绿色！

这种方式增加了一个额外的标签而且还要配合zoom:1才能达到效果,在父容器设置过width height position(relative absolute) 属性的时候不需要配合zoom:1。

三、使用overflow .cd方法闭合浮动

使overflow属性来清除浮动有一点需要注意，overflow属性共有三个属性值：hidden,auto,visible。可以使用hiddent和auto值来清除浮动，但不要使用visible值(visible只对IE浏览器有效)。基于IE 6 的haslayout 问题，我们定义一个单独的class应用于父元素而闭合浮动(使用auto可能会产生滚动条)。

将该class用于内部有浮动的父元素就行了。

IE6 BUG诡异的行内替换元素设置浮动并设置text-indent：xx px 影响相邻元素布局

E6 BUG诡异的行内替换元素设置浮动并设置text-indent：xx px 影响相邻元素布局

一个奇怪的现象：一个DIV里面有两个各元素，一个元素右浮动，浮动的元素设置了个text-indent,而另外一个元素却仅在IE6下产生了和该元素一样大小的类padding-left间距值。

发现原型：

<input type=“button” value=“搜索” /> <a href=“#”>啊啊啊啊</a></div>
<span style=“color: #ff0000;”>在input标签中设置text-indent:20px后，会发现 a标签 在IE6中会出现一个类似左padding值等于20px。</span>

经过一系列测试发现问题产生的条件：

1、 一个内联替代元素【input,img等】和一个内联元素【含内联可替换和内联不可替换元素】相邻

2、 内联可替代元素浮动，内联元素不浮动

3、 内联可替代元素设置 text-align: xx px;

导致的结果：内联元素出现类似padding-left:xx px 的左距离 其中xx 等于 内联可替代元素的text-indent数值。

已知解决方式：

1、 将内联元素也设置浮动

2、 后一元素使用绝对定位

3、 用padding-left替代text-align

未知求解：未知其产生BUG的原理从而找出根本的解决方式，上述解决方式个人觉得较为牵强。

以上探究纯属个人测试个人观点，如有不对欢迎指正。
